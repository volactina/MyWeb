## PROMPTING.md（省 token 工作流模板）

这份文档的目标是：在 Cursor 里让每次请求携带的上下文尽量小、回合尽量少，从而显著降低 token 消耗，同时让改动更稳定可控。

## 核心原则（优先级从高到低）

- **短会话**：一个小功能 / 一个 bug 一个对话，做完就开新 chat（避免携带超长历史）。
- **少上下文**：每次只指定 **1–2 个文件**，并贴出 **20–80 行**相关片段（不要让系统“自动找一堆相关文件”）。
- **少日志**：只贴错误堆栈/日志的 **最后 30–120 行**（信息密度最高）。
- **少回合**：一次说清：目标、范围、约束、验收方式，减少来回追问。
- **少大段输出**：让助手“只给最小补丁/最小改动”，不要输出整文件或超长代码块。

## 本项目的“建议改动入口”（避免读全仓库）

- **后端**（优先改）
  - `backend/routes.py`：API 路由与参数校验（不涉及业务规则时只改这里）
  - `backend/service_items.py`：父子同步、阻塞规则、path 计算、写入前归一化等核心业务
  - `backend/normalize.py`：状态/ID 列表解析、时间戳、字段缺省等通用逻辑
- **前端**
  - `static/app.js`：渲染、弹窗、交互（尽量只改相关函数附近）
  - `templates/index.html`：样式与 DOM（只改需要的 CSS/结构片段）

## 模板 A：最省 token 的“单点修改”

适合：改一个按钮、一个样式、一个小逻辑、一个字段显示。

复制后填空：

```
目标：<一句话说明要改什么>
范围：只改 @<file1> @<file2>（不要读其他文件）
位置：下面是相关片段（请只在这块附近改，不要重构全文件）
约束：不改接口/不引入新依赖/不改变其它行为
验收：<点哪里/跑什么命令/看到什么结果>

<贴 20–80 行代码片段>
```

建议加一句“输出要求”（更省 token）：

```
输出：只给最小改动补丁，不要贴整文件。
```

## 模板 B：最省 token 的“修 bug / 报错”

适合：TemplateNotFound、JS 报错、接口 400/500、按钮没反应等。

```
问题：<一句话>
复现：<1-3 步>
期望：<一句话>
实际：<一句话>
范围：只允许改 @<file1>（必要时再加 @<file2>）
证据：
1) 错误堆栈最后 30–120 行
2) 相关代码 30–120 行
输出：只给最小改动补丁 + 3 条验证步骤
```

## 模板 C：小功能但会改多文件（仍然省 token）

适合：要改前后端多个文件，但范围仍然可控。

```
功能：<一句话>
必须包含：
- <要点1>
- <要点2>
- <要点3>
不做：
- <明确排除项>
涉及文件：@<file1> @<file2> @<file3>
顺序：先后端再前端；最后给我 3 条最小测试清单
输出：每个文件只给最小改动补丁，不要贴整文件
```

## 通用“省 token”开场一句话（强烈推荐）

每次开新对话，直接用这一句做开头：

```
只改 @<file1> @<file2>，只围绕我贴的代码片段修改，别读其他文件；不要输出大段代码，只给最小补丁与最小测试步骤。
```

## 常见“token 暴涨”触发器（尽量避免）

- 把整份文件/整屏日志贴进对话
- 让助手“通读项目/搜索全仓库”
- 长对话不断续聊（历史会越来越大）
- 反复让助手输出大段 diff / 大段代码（后续请求又会把这些内容带入）
